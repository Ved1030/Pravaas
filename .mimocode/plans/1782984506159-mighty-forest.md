### Step 2: Config Files

**Create `backend/src/config/environment.js`:**
```js
require('dotenv').config();
module.exports = {
  PORT: process.env.PORT || 10000,
  NODE_ENV: process.env.NODE_ENV || 'development',
  CLIENT_URL: process.env.CLIENT_URL || 'http://localhost:5173',
  SARVAM_API_KEY: process.env.SARVAM_API_KEY || '',
};
```

**Create `backend/src/config/cors.js`:**
```js
const env = require('./environment');
const whitelist = [env.CLIENT_URL, 'http://localhost:5173', 'http://localhost:3000'];
module.exports = {
  origin: (origin, cb) => {
    if (!origin || whitelist.includes(origin)) cb(null, true);
    else cb(new Error('Not allowed by CORS'));
  },
  credentials: true,
};
```

**Create `backend/src/config/server.js`:**
Central server setup — loads helmet, compression, morgan, rate-limit, hpp, cors, and mounts them on the app.

### Step 7: Standardized Response Format

Create `backend/src/utils/response.js`:
```js
exports.success = (res, message, data, statusCode = 200) => {
  res.status(statusCode).json({ success: true, message, data });
};
exports.error = (res, message, error, statusCode = 500) => {
  res.status(statusCode).json({ success: false, message, error });
};
```

**IMPORTANT**: Only use this for NEW responses (health checks, 404, validation errors). Do NOT change existing controller response formats — the user explicitly said "DO NOT change any existing API responses unless required". Existing controllers will keep their current `res.json(...)` patterns.

### Step 8: Input Validation

### Step 13: Frontend API URL Update

Update `frontend/src/lib/api.ts`:
- Change `BASE_URL` from `http://localhost:5000/api` to use the Render backend URL
- Since it's deployed on Render, use env variable or hardcoded Render URL
- Add `/v1` prefix to all paths

**Note**: The frontend BASE_URL should ideally come from an env var. For Vercel deployment, set `VITE_API_URL` in Vercel env vars.

Update `api.ts`:
```ts
const BASE_URL = import.meta.env.VITE_API_URL || 'http://localhost:5000/api/v1';
```

Then all existing fetch calls like `${BASE_URL}/route/plan` already resolve to `/api/v1/route/plan`. The paths in the fetch calls use relative paths from `/api`, so changing the base to include `/v1` works.

Wait — current paths are like `${BASE_URL}/route/plan` where BASE_URL is `http://localhost:5000/api`. So the full URL is `/api/route/plan`. If I change BASE_URL to `http://localhost:5000/api/v1`, the full URL becomes `/api/v1/route/plan`. That's correct.

### Step 14: Backward Compatibility Routes

To NOT break existing frontend during transition, add backward-compat redirects in `app.js`:
```js

```

Or just mount the v1 router at both `/api/v1` and `/api` temporarily.
